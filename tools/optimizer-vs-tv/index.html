// WHY OPTIMIZER COUNTS MAY DIFFER FROM TRADINGVIEW'S STRATEGY TESTER

When you compare ORBAlpha Optimizer results to a TradingView Strategy Tester run on the same configuration, you may notice the Optimizer reports more trades and a different cumulative PnL. This is expected, documented, and the Optimizer's numbers are the ones that match what your live trading system will execute.

This page explains why.

// WHAT YOU'LL SEE

Run a backtest on TradingView's Strategy Tester. Note the trade count and total PnL.

Run the same configuration through the ORBAlpha Optimizer. The Optimizer's trade count will typically be higher by a small number of trades — and those extra trades cluster at the start of the test window.

For example, a representative MES configuration over a 90-day window might show:

SourceTradesPnL
TradingView Strategy Tester display44$3,553
ORBAlpha Optimizer49$4,264
Pine Script's actual internal trade count49$4,264

The 5-trade gap is not an Optimizer error. It's the difference between what Pine Script actually does and what TradingView's Strategy Tester shows you.

// WHY THIS HAPPENS

TradingView's Strategy Tester applies an undocumented warmup filter to its trade list display. This filter suppresses the display of early trades that Pine Script's strategy code actually fires.

The mechanism is rooted in how Pine Script accesses higher-timeframe data. Strategies like ORB/ORF use daily-bar lookbacks (e.g., the previous 5 days' high to compute a stretch range). Pine Script accesses these via request.security() calls.

By default in Pine v5+, request.security() uses barmerge.lookahead_off semantics. From TradingView's official Pine Script documentation:

"This method returns data that is up to one HTF bar 'late', and is thus not subject to change."

In practice: a 5-day stretch lookback functionally requires 6 daily bars to produce a fully populated value at the start of a backtest window. Pine Script's broker emulator (the engine that simulates trades) does not wait for this warmup — it fires entry orders as soon as conditions evaluate true. TradingView's Strategy Tester display, however, suppresses those early trades.

The result: Pine fires N trades. TV displays N − k trades, where k depends on the strategy's lookback parameters.

// THE OPTIMIZER MATCHES PINE'S ACTUAL BEHAVIOR

The ORBAlpha Optimizer engine is a JavaScript port of the v2.3.2 Pine indicator. We've verified its logical parity by:

  1. Trade-by-trade comparison against Pine Script's strategy.closedtrades count and per-trade log.info() output.
  2. Cross-asset confirmation on MES, MGC, and MCL — engine and Pine fire the same trades, with prices and PnL matching to the cent.
  3. Diagnostic Pine builds (v2.3.3-TEST and v2.3.4-TEST) that surface Pine's internal trade list directly, bypassing TV's display filter.

In every case, the Optimizer's output matches Pine's actual fired trades — not TV's filtered display.

// WHY WE DON'T “FIX” THE OPTIMIZER TO MATCH TV

Calibrating the Optimizer to mirror TV's display would mean deliberately hiding trades that your live system will execute. That's the opposite of what a backtesting tool should do.

The Optimizer is calibrated to reality, not to TV's display.

Specifically:

  • Pine's broker emulator fires N trades.
  • Live webhook execution via TradersPost fires the same N trades.
  • TV's Strategy Tester display shows N − k.
  • The Optimizer reports N.

If we suppressed those k trades to match TV, your Optimizer numbers would be less predictive of live trading outcomes. Configurations that look acceptable in TV's display might be carrying real PnL in those hidden early trades — PnL that your live system will realize. We'd rather show you the truth.

// LIVE TRADING IMPLICATIONS

If you trade ORBAlpha strategies via webhooks (TradersPost or equivalent), your fills will match Pine's actual strategy.closedtrades — which is what the Optimizer reports.

This means:

  • Optimizer PnL projections are the right reference for live performance.
  • The early trades hidden by TV's display will fire in live trading.
  • Comparing live PnL against TV's display will show consistent overperformance early in any window.
  • Comparing live PnL against the Optimizer should show close alignment.

If you want a single number to use as your live-trading expectation, use the Optimizer's number, not TV's display.
